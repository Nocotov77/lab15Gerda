## Теоретическая часть

Apache Kafka — это распределённая система потоковой обработки, работающая по модели издатель‑подписчик. Основные понятия:

- **Топик (topic)** – категория, в которую публикуются сообщения. В лабораторной работе топик `orders` содержит события о заказах.
- **Партиция (partition)** – часть топика, обеспечивающая параллелизм и горизонтальное масштабирование. Каждое сообщение внутри партиции имеет уникальный номер — **оффсет (offset)**.
- **Продюсер (producer)** – приложение, отправляющее сообщения в топик. Может задавать ключ сообщения для управления партиционированием.
- **Консюмер (consumer)** – приложение, читающее сообщения из топиков. Консюмеры объединяются в группы (`group_id`): внутри одной группы партиции распределяются между экземплярами, что позволяет обрабатывать данные параллельно и гарантирует однократную доставку каждого сообщения группе.
- **Потоковая обработка** – обработка данных «на лету» с минимальной задержкой, в отличие от пакетной ETL, где данные накапливаются и обрабатываются периодически.

Для развёртывания Kafka используется Docker Compose, поднимающий сервисы Zookeeper (координатор), Kafka‑брокер и веб‑интерфейс Kafka UI.

## Ответы на контрольные вопросы

1. **Что такое топик, партиция, офсет в Kafka? Как они использовались в работе?**
   - Топик — логическая группа сообщений (в работе `orders`). Партиция — физический сегмент топика для параллельной обработки (создано 3). Офсет — уникальный номер сообщения в партиции (использовался для отслеживания позиции консюмера и контроля дубликатов).

2. **В чём разница между продюсером и консюмером? Зачем нужна группа консюмеров (`group_id`)?**
   - Продюсер публикует сообщения, консюмер подписывается и обрабатывает их. Группа консюмеров гарантирует, что каждое сообщение из партиции будет доставлено только одному экземпляру внутри группы, что позволяет масштабировать обработку и избегать дублирования.

3. **Какие преимущества даёт потоковая обработка перед пакетной (ETL из Части 1)?**
   - Минимальная задержка (миллисекунды против минут/часов), возможность немедленной реакции на события (алерты, уведомления), горизонтальная масштабируемость за счёт партиций, высокая отказоустойчивость.

4. **Приведите 3 реальных сценария использования Kafka в бизнесе (не связанных с логированием).**
   - Обработка заказов интернет-магазина в реальном времени (обновление стоков, уведомления клиентам).
   - Детектирование мошеннических транзакций в банковской сфере (анализ потока транзакций на подозрительную активность).
   - Агрегация данных IoT с промышленных датчиков для предиктивного обслуживания и аварийных оповещений.

5. **Что произойдёт, если запустить 2 консюмера с одинаковым `group_id`?**
   - Партиции топика будут автоматически распределены между ними. Каждый консюмер получит только часть сообщений (те партиции, которые ему назначены), что увеличит общую пропускную способность. Если партиций меньше, чем консюмеров, лишние будут простаивать.
